## 1. 修复指针解引用无法修改栈变量

### 问题表现

示例 06 中执行以下代码后，变量值没有改变：

```cpp
int x = 42;
int* p = &x;

*p = 99;
```

界面仍显示 `x = 42`。将指针改为指向 `y` 后执行 `*p = 200`，也存在相同问题。

### 原因

模拟器原先只在 `_heap` 中查找解引用目标：

```python
if ptr_val in self._heap:
    self._heap[ptr_val].value = rhs
```

`x` 和 `y` 是局部变量，保存在模拟栈帧中，并不在堆映射里。因此，指针虽然保存了正确的栈地址，但解引用写入被跳过，也没有生成对应的执行快照。

### 修复内容

文件：`src/cpp_interpreter.py`

- 新增按模拟地址查找当前存活栈变量和全局变量的逻辑。
- `*p = value` 现在支持修改栈变量、全局变量和堆变量。
- `value = *p` 现在支持读取栈变量、全局变量和堆变量。
- 解引用写入快照同时高亮指针和被修改的目标变量。

### 验证结果

示例 06 输出正确：

```text
x = 42
x after *p=99: 99
y after *p=200: 200
```

同时验证了全局变量、堆变量的指针写入，以及栈变量的解引用读取。

## 2. 修复示例 07 递归执行和栈帧展示

### 问题表现

示例 07 的 `factorial(4)` 原先直接返回 `1`：

```text
4! = 1
```

界面也只显示一层 `factorial` 栈帧，没有展示递归调用过程。

### 原因

赋值右侧的函数调用使用以下执行路径：

```text
_eval_expr()
  -> _eval_function_call()
      -> _execute_function_body()
```

`_execute_function_body()` 原先没有处理 `if (n <= 1)`。执行 `factorial(4)` 时，条件语句被跳过，下一行 `return 1` 被当成无条件返回，因此：

- `factorial(n - 1)` 不会执行。
- `sub` 和 `result` 不会计算。
- 不会创建多层递归栈帧。

### 修复内容

文件：`src/cpp_interpreter.py`

- 为返回值函数体执行器增加 `if` 代码块处理。
- 使用明确的 `(did_return, value)` 语义传播 `return` 状态。
- `if` 块中的 `return` 会立即结束当前函数，而不是只结束局部代码块。
- 递归调用的每一层使用独立的 `StackFrame`。
- 继续由 `_eval_function_call()` 负责当前调用的栈帧弹出，避免重复弹栈。
- 函数实参先在调用者作用域求值，避免递归调用时变量遮蔽。
- 函数调用快照使用实际调用行号：主调用定位到第 16 行，递归调用定位到第 9 行。

### 正确的调用栈

执行到最深处时，栈帧为：

```text
main
factorial(4)
factorial(3)
factorial(2)
factorial(1)
```

### 正确的返回过程

```text
factorial(1) -> 1
factorial(2) -> 2
factorial(3) -> 6
factorial(4) -> 24
```

最终输出：

```text
4! = 24
```

- Python 语法检查通过。
- 示例 07 输出、递归深度和返回值正确。
- 示例 03 的带返回值函数调用回归通过。
- 示例 08 的独立函数调用回归通过。
- 全部 19 个内置 C++ 示例均可完成解析执行。

## 3. 修复小屏幕多栏内存区适配

### 问题表现

在小屏幕或窄窗口中，分开多栏布局会把窗口撑出屏幕，部分内存栏位被裁切且无法访问。详细模式尤其明显。

### 原因

文件：`src/main_window.py`

界面同时存在多层硬最小宽度：

- 主窗口最小尺寸为 `1280x780`。
- 左侧区域最小宽度为 `420px`。
- 右侧区域最小宽度为 `900px`。
- 每个内存栏位最小宽度为 `300px`。
- 切换详细模式时，右侧最小宽度被错误设置为 `1800px`，但详细模式实际为上下两行、每行三列。
- 多栏整体没有水平滚动容器，栏位内部又关闭了水平滚动。

### 修复内容

- 主窗口最小尺寸调整为 `960x620`。
- 左侧区域最小宽度调整为 `360px`。
- 移除右侧区域和模式切换时的硬最小宽度。
- 为 split 多栏整体增加统一的水平滚动区域。
- 每个栏位继续独立负责垂直滚动，并关闭内层水平滚动。
- standard 保持三列，simple 保持四列。
- detailed 保持 `2x3` 布局，上下两行通过同一个水平滚动区域同步移动。
- unified 单竖列布局保持不变。

### 验证结果

离屏 Qt 验证结果：

```text
standard: 内容宽度 788px，3 个内层面板
simple:   内容宽度 1052px，4 个内层面板
detailed: 内容宽度 788px，6 个内层面板
```

在 `500px` 可视宽度下，三种模式都会产生正确的整组水平滚动范围。另已验证：

- `960x620` 窗口尺寸。
- simple、standard、detailed 来回切换。
- split 与 unified 切换。
- 示例 07 执行和面板刷新。
- Python 语法检查。

## 4. 修复堆内存增长方向展示

### 问题表现

单竖列布局顶部标记为高地址、底部标记为低地址，但堆卡片按地址升序从上到下显示，首分配的低地址块位于上方。堆标题还显示为 `↓ 向高地址增长`，与地址标尺相反。

split 堆面板也存在相同的排序问题：虽然卡片组贴近底部，但卡片内部仍是低地址在上、高地址在下。

### 原因

文件：`src/panels.py`

- `UnifiedMemoryPanel.refresh()` 对堆变量按地址升序渲染。
- unified 的 heap section 使用 `AlignTop`。
- 三种 unified 分段模式的堆标题箭头方向错误。
- `MemoryPanel.refresh()` 在 `grow_up=True` 时仍然按地址升序渲染。

堆地址分配器本身没有问题，地址一直从 `0x01000000` 向高地址递增。

### 修复内容

- unified 的 heap section 改为底部对齐。
- unified 堆卡片改为按地址降序显示，高地址在上、低地址在下。
- simple、standard、detailed 的堆标题统一改为 `↑ 向高地址增长`。
- split 的 `MemoryPanel(grow_up=True)` 同样改为地址降序。
- 其他内存区域的排序保持不变。

正确的视觉顺序为：

```text
顶部：高地址 / 后分配
0x01000008
0x01000004
0x01000000
底部：低地址 / 先分配
```

首个低地址块位于堆区底部，后续高地址块依次出现在上方。

### 验证结果

- split 和 unified 均按地址降序显示堆卡片。
- simple、standard、detailed 的 unified heap section 均为底部对齐。
- 示例 13 的 15 个含堆内存执行步骤全部通过排序验证。
- 分配和释放后，剩余堆块仍保持正确地址顺序。
- Python 语法检查通过。
